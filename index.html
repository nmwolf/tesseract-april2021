Overview¶

This session is dedicated to exploring the tricks and tools available to build a workflow that turns digital images of text into computer-readable text. Its focus is on skills needed for an individual who is trying to bring together a corpus of texts for the purposes of text analysis, a website, a Digital Humanities project, or a small-scale digital library.

We'll focus on the following goals:

  • Review a few options for making the digital capture of a text.
  • Walk through some solutions for bulk transformation of image file types into formats that are OCR-ready
  • Take a look at using Tesseract 4 for OCR, using a few different languages as examples
  • Examine the HOCR output structure
  • [OPTIONAL] Open hour to talk through Tesseract 4 installation and Tesseract's training manual

OCR Options and Tesseract 4¶

There are increasingly more options for turning bitmap images of texts into computer-readable text. Even out-of-the-box document scanning tools will try for this, and Adobe's DC Reader, if prompted to attempt to read a text, will do pretty well in making a text out of an image. Not all of these will succeed on a broad range of languages, fonts, or document qualities, however, and many are designed for very modern texts in order to make documents accessible for screen readers.

From a pedagogical standpoint, ABBYY Finereader is a far better tool for teaching OCR (and especially training one) because of its graphical interface. But in an all-remote teaching environment ABBYY Finereader is not ideal because it requires a paid license and optimally runs on Windows (the Mac version is not very good). So for this session we need an open-source alternative.

Tesseract, while much more difficult to use, is a really robust OCR tool with support for 4500 fonts and ~130 languages, trained on 400k lines of text. It is also free, thus more suitable for session participants to install. But training the latest Tesseract 4 is a challenge in a way it was not for Tesseract 3 or ABBYY. A 4-core configuration is recommended to accommodate an LSTM machine learning approach, and its instruction materials are largely oriented around a Linux environment. Hence, the lesson below cannot viably focus on how to train Tesseract 4 on new fonts or languages. Instead, we'll reserve time at the end of the session to share stories.

2. First Things: When Do You Need OCR?¶

Or rather, when do you not need OCR? An image file that contains embedded encoded text or a PDF that presents encoded text and not a bitmap image of text, does not need to be subject to OCR to turn it into the text needed for search or analysis. In most PDF readers such as Adobe DC, you can open such a PDF and select the option to export it to plain text, HTML, or another format:

Many common command-line utilities related to PDFs also rely on this transformation available for non-bitmap representations to make a conversion. A common one is XpdfReader (pdftotext), which can be installed and used to turn such PDFs quickly into plain text.

Here's an example using the UN Universal Declaration of Human Rights, found in an 8-page PDF format on the UN website:

Principles for Image Captures¶

Deskew is possible using image editors. But problems like this are deadly:

Source: University of Washington Accessible Technology Resources, https://www.washington.edu/accessibility

Parameters often stated:

  • 300 dpi minimum (sometimes, as with ABBYY, greater than 300 dpi not required)
  • If using a phone, keep plane of camera parallel to page
  • Avoid capture of text/characters outside of current page

See also this taxonomy of image-causing OCR problems from Tesseract: https://tesseract-ocr.github.io/tessdoc/ImproveQuality.html

...or, toss some other (preferably open-source) image processing options in the chat!

4. Image Preprocessing¶

Typical steps include:

  • Apply grayscale (remove color from image)
  • Binarization (reduce all pixels to black or white)
  • Filter (e.g. of faint background shading)
  • Rotate / deskew
  • Text smoothing (via thresholding or blur)

A great tool is the OS-agnostic Fred's Image Magick textcleaner utility: http://www.fmwconcepts.com/imagemagick/textcleaner

5.1 Language Parameter¶

Adding the -l flag allows you to explicitly state the language or languages using the ISO 639-2 three-digit language codes. For example:

-l spa

or

-l eng+spa

A code list (from the Library of Congress) can be found here: https://www.loc.gov/standards/iso639-2/php/code_list.php

5.2 Page Segmentation¶

State explicitly how to segment the file being, i.e. whether it is a page of text, a column of text, a single character, etc. OSD is Orientation and Script Detection

tesseract INPUTFILE OUTPUTFILE --psm 1

5.3 OEM (Important)¶

The latest Tesseract (v 4, and a beta v 5) have two different engines. The older is a classic OCR predictive approach, the one implemented in Tesseract 3 and ealier. The other is an unsupervised machine learning approach (the characteristics, and thereby the identity) of the characters are inferred from repeated investigation of their characteristics (the LSTM model).

It is my understanding that the default behavior for Tesseract 4 is a combination of classic and LSTM, based on the notion that the default is to use all engines for which training data has been installed, and if the default for current installation is for both types of training data to be included, it would follow that oem = 2 is the expected base behavior. See options listed below.

See this 2018 discussion of default oem settings on Stack Overflow.

Example:

tesseract INPUTFILE OUTPUTFILE --oem 1

7. The HOCR Output¶

An HOCR file is an HTML-based means of describing an OCRd output matched to a single page/document. An outline of the specs are here: http://kba.cloud/hocr-spec/1.2/

In particular, let's focus on the BBOX property: http://kba.cloud/hocr-spec/1.2/#bbox

In Tesseract, we can specify that an output be given in HOCR by including it as a parameter:

In [17]:
%%bash

tesseract wpa-lifehistory-swenson-1938.tif wpa-lifehistory-text-output hocr

Every word, line, and page area has its own bounding box. We can write some quick code in Python to visualize this.

The approach below is to treat each line bounding box coordinates as an array. We then use the Python PIL library to overlay boxes over the base image:

In [55]:
import numpy as np
from bs4 import BeautifulSoup
from PIL import Image, ImageOps, ImageDraw
from IPython.display import display



def load_hocr_lines(filepath):
    """
    Page Array Structure
    col 0 = ID number of line
    col 1-4 = bbbox x1, y1, x2, y2
    """
    page_array = []
    rawhtml = BeautifulSoup(open(filepath, encoding='utf-8'), "lxml")
    for line in rawhtml.html.body.div.find_all('span'):
        line_list = []
        if line['class'][0] == 'ocr_line':
            line_list.append(int(line['id'].split('_')[2]))
            line_list += [int(i) for i in line['title'].split(';')[0].split(' ')[1:]]
            page_array.append(line_list)
    return np.array(page_array)


def imagebuilder(page_array, image_filename):

    pageimg = Image.open(image_filename).convert('RGB')
    overlay = ImageDraw.Draw(pageimg, 'RGBA')

    # Draw the hocr boxes
    
    for i in range(len(page_array)):
        
        overlay.polygon([(page_array[i, 1], page_array[i, 4]),
                         (page_array[i, 1], page_array[i, 2]),
                         (page_array[i, 3], page_array[i, 2]),
                         (page_array[i, 3], page_array[i, 4])],
                         fill=(255, 0, 0, 90))
        

    display(pageimg)
